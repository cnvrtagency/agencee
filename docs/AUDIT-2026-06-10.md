# Agencee — Full System Audit (10 June 2026)

Read-only diagnostic audit. No code was changed. Five parallel audit passes covered: (1) content/publishing + GitHub, (2) GSC + keywords, (3) jobs + multi-tenancy + scalability, (4) dashboard + Ada, (5) notifications + environment.

---

## Executive summary

The core single-client loop — Ada writes → commits to main → Vercel preview → approve → promote to production — is wired end to end and functional, as are GSC sync, keyword discovery/approval, briefing items, notifications scaffolding, and the dashboard (all real data). But the system is **not production-ready as a multi-tenant SaaS**: three API routes let any caller read or trigger another workspace's data, the cron endpoint is unauthenticated because `CRON_SECRET` is unset and the check fails open, email notifications silently never send (`RESEND_API_KEY` missing), and the connections publish routes will crash (`SUPABASE_SERVICE_ROLE_KEY` missing). It is also **not ready for a second client**: `hearbetternow.co.uk` is hardcoded in three places, Ada's blog format is Hear Better–specific in the system prompt, local-intent detection is hardcoded to North East England, and `full_autopilot` calls the deprecated merge route. The biggest risks are workspace isolation gaps (security) and the pervasive silent-failure pattern.

## Section A — Content creation and publishing flow

### A1. `append_blog_post` step by step (`src/app/(app)/agents/[id]/page.tsx` ~620-748)

1. Client lookup by name (line 621)
2. Extract slug from `post_tsx` (line 625)
3. Strip `as const` directives (lines 631-633)
4. `PUT /api/github` → write `next-public/content/blogPosts/[slug].tsx` to production branch (lines 651-660)
5. Read blog index `next-public/content/blogContent.tsx` (lines 664-668)
6. Commit updated index with import + registration (lines 670-682)
7. Insert `content_outputs` row, `approved: false` (lines 705-714)
8. Insert `content_history` row (lines 716-724)
9. Fire `POST /api/notifications/output-ready` (lines 728-742)

Files modified: 2 GitHub files. DB rows: `content_outputs` first, then `content_history`. **Both inserted AFTER the GitHub commits.**

### A2-A3. Branch/repo

Dynamic per client: `client_profiles.github_repo`, `github_branch || 'main'` (`/api/github/route.ts` lines 76, 109, 167). Not hardcoded. The directory prefix `next-public/content/` IS hardcoded.

### A4. Format

TypeScript object literals (`BlogPost` exports), one file per post. Not markdown/MDX/JSON.

### A5. Vercel build trigger

Every commit to main triggers a Vercel build. Nothing in code prevents production deployment — the preview→production gate depends entirely on the user having disabled auto-promotion in Vercel project settings.

### A6. Complete path to live

1. Ada calls `append_blog_post`
2. Files committed to production branch via PUT /api/github
3. `content_outputs` row inserted (`approved: false`)
4. `content_history` row inserted
5. Notification fires
6. **Manual:** user opens `/outputs/[id]`
7. User clicks "Approve and publish"
8. `POST /api/vercel/promote`: finds latest READY preview deployment → Vercel promote API → sets `published_url`, `approved: true`
9. Live

### A7. DB-after-GitHub risk

If GitHub succeeds but the Supabase insert fails, Ada's tool response (line 746) reports success anyway. Content exists on GitHub but is orphaned — user can never find it to approve. **Silent failure.**

### A8. Branch written

Production branch (`github_branch || 'main'`). Draft branches fully removed. AGENCEE.md line ~734 still mentions `agencee-drafts-[client-slug]` — stale docs.

### A9. "Approve and publish" trace (`/outputs/[id]/page.tsx` lines 142-173, button line 329)

1. Update `content_outputs.approved = true` (line 147)
2. `POST /api/vercel/promote` (line 148)
3. Promote route: fetch 5 most recent preview deployments → first READY → POST Vercel v13 `/deployments/{uid}/promote`
4. Update `published_url` (hardcoded `https://hearbetternow.co.uk` — line 59 of promote route)
5. Insert `content_history` (lines 162-168)
6. Show "Live"

### A10. Vercel promote route failure modes

- Missing VERCEL_TOKEN/VERCEL_PROJECT_ID → 500
- No READY deployment → 404
- Promote API failure → 400 with message

Env: `VERCEL_TOKEN` and `VERCEL_PROJECT_ID` ARE set, but **duplicated** in `.env.local` — empty entries (lines 15-17) followed by real values (lines 18-19). Last wins; fragile.

### A11. Validation

**No `validateBlogPost` function exists anywhere.** Only regex cleaning. No validation of slugs, required fields, arrays, TypeScript syntax, or structure. Malformed TSX has already broken the client build twice (`as const` incident, `title:`/`label:` incident).

### A12. `cleanContent` (`src/lib/content-clean.ts`)

Does: em dash → comma, smart quotes → straight, collapse 3+ newlines, `relatedServices` `title:` → `label:`.

Applied: `append_blog_post` (line ~710, before DB insert), `write_file` (line ~979), `/api/run-task` (line 124), output edit save (`/outputs/[id]/page.tsx` line ~207).

**Applied before DB save only — NOT before the GitHub commit.** GitHub receives raw content.

### C20. Column check

Schema columns confirmed: `id, client_id, keyword..., google_account_email, property_url, access_token, refresh_token, token_expires_at, connected_at, last_synced_at, status, workspace_id`.

### C21. `getValidAccessToken` (`src/lib/gsc.ts:11-36`) — WORKING

60-second expiry buffer; decrypts cached token if valid; otherwise calls `/api/auth/google/refresh` which exchanges refresh_token, re-encrypts, updates DB, returns fresh token.

### C22. Sync return values

Success: `{ synced: {7d,28d,90d}, sitemap_urls, property, date_range, page_rows_synced, briefing_items }`.
Failures: 400 connection param missing, 404 not found, 400 token refresh failed, 500 with stack.

What sync does: 3 parallel periods (query+page rows ×1000 + no-dimension `__total__`) → delete + reinsert `search_performance` → update `keyword_banks.current_position` and `content_history.ranking_position` → populate `content_performance` → `discoverKeywordsFromGSC` (deduped union of all periods) → briefing items → sitemap fetch → `last_synced_at`.

### C23. GSC in system prompt

NOT baked in — fetched on demand via `analyse_gsc` tool. Proactive instruction at `agents/[id]/page.tsx` lines 111-119 (call before content recommendations when client context exists; conversational exception). Correct lean pattern.

### C24. Keyword suggestions from GSC

## Section D — Keyword suggestions flow

### D26. Approve trace — WORKING

Button (`keywords/page.tsx:229`) → `approve()` (98-116) → `POST /api/keywords/approve` → load suggestion → infer intent (`informational` if interrogative regex; else `commercial`) and funnel (`top`/`bottom` if local/`middle`) → opportunity_score from metadata position (≤10→60, ≤20→75, else 50) → insert `keyword_banks` (workspace_id, client_id, keyword, intent, funnel_stage, priority 5, opportunity_score, + monthly_volume/current_position when present) → update suggestion `status='approved'` → fadeOut + toast.

### D27. Service key vs anon

Service key (`/api/keywords/approve/route.ts:4-8`). Required: server-side routes have no user session, so anon-key inserts are blocked by RLS. Route explicitly carries `workspace_id` from the suggestion. Correct pattern.

### D28. Reject flow — WORKING

Reject button → reason input → `POST /api/keywords/reject` → merge `reject_reason` into `metadata` JSONB, `status='rejected'`. Rejected tab queries `.eq('status','rejected')` with client filter. Un-reject button restores to pending.

### D29. Rejected exclusion — WORKING

`gsc-keywords.ts:10-20` loads `keyword_banks` + pending + rejected suggestions into one lowercase Set; line 35 filters candidates against it. Rejected keywords are never re-proposed.

### D30. Duplicate check — WORKING

Case-insensitive Set over `keyword_banks.keyword` for the client. "Hearing Aids" vs "hearing aids" both normalise → duplicate excluded.
